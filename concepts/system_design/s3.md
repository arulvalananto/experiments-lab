# Object Storage Isn’t Just “Upload a File, Get a URL Back”

That idea works for small side projects. It breaks completely at massive scale.

Amazon S3 launched in 2006. By 2013, it stored 2 trillion objects. By 2021, it crossed 100 trillion. At that scale, the system no longer resembles a file server or relational database. It becomes a completely different class of infrastructure, built around different tradeoffs.

This kind of problem appears in senior and staff-level system design interviews at companies building storage-heavy systems.

It tests whether you can:

- Understand why different storage systems exist
- Design for durability when hardware failures are constant
- Balance consistency, cost, and performance

We’ll build this step by step:

- What object storage is
- How it differs from other storage types
- Requirements and capacity estimation
- High-level architecture
- Disk persistence
- Durability strategies
- Metadata design
- Versioning
- Large file uploads
- Garbage collection

At each step, the focus is on *why* decisions are made.

## What Is Object Storage?

Before designing anything, we need clarity on what object storage actually is.

Engineers often confuse the three main storage types. Each exists for a reason, and choosing the wrong one is expensive to fix later.

---

## Block Storage

Block storage is the lowest-level abstraction.

A disk (HDD or SSD) is exposed as raw blocks, usually 4KB each. The operating system decides how to organize these blocks using a file system. Some systems, like databases or VM engines, skip the file system and manage blocks directly for better control and performance.

Block storage can also be network-based:

- Fibre Channel
- iSCSI

Cloud providers expose this as services like EBS, where a network volume behaves like a local disk.

**Pros:**

- High performance
- Full control

**Cons:**

- Expensive
- Hard to scale to very large sizes

---

## File Storage

File storage builds on top of block storage.

It introduces:

- Files
- Directories
- Hierarchical structure

You interact with it using paths like:
`/documents/report.pdf`

It supports shared access across machines using:

- NFS
- SMB/CIFS

**Pros:**

- Easy to use
- Supports shared access

**Cons:**

- Poor scalability at billions of files
- Directory traversal becomes slow and complex

---

## Object Storage

Object storage is fundamentally different.

It makes a key tradeoff:

Give up mutability and some performance in exchange for massive scalability, high durability, and low cost.

Key characteristics:

- Flat namespace (no real directories)
- Data stored as objects inside buckets
- Accessed via HTTP APIs
- Objects are immutable

You cannot partially update an object:

- Replace the entire object, or
- Create a new version

This immutability simplifies replication and consistency, which is why object storage scales so well.

Examples:

- Amazon S3
- Google Cloud Storage
- Azure Blob Storage

Use cases:

- Media storage
- Backups
- Data lakes
- Machine learning datasets
- Static assets

---

## Core Constraint

Objects are immutable.

You cannot edit part of an object. You either replace it entirely or create a new version.

This constraint is intentional and enables scalability and durability.

---

## Key Concepts

### Bucket

- Logical container for objects
- Not truly hierarchical
- Globally unique name
- Must exist before storing objects

### Object

An object contains:

- Payload → actual data
- Metadata → descriptive key-value pairs

Metadata is stored separately and is much smaller.

### Object Key

- Unique identifier (string)
- Looks like a file path but isn’t hierarchical

Example:
`photos/2024/vacation.jpg`

This is just a string, not a directory structure.

### Versioning

- Keeps previous versions instead of overwriting
- Enables recovery and auditing

### Durability SLA

Example: 99.999999999% (eleven nines)

This means extremely low probability of data loss, achieved through replication and redundancy.

## Clarifying Requirements

Before designing, define the system clearly.

Skipping this leads to incorrect designs.

## Questions to Ask

- What operations are required? (upload, download, delete, list)
- Is versioning needed?
- Expected data size?
- Durability and availability targets?
- Large file support?
- Access control requirements?

## Assumptions

- Operations:
  - Bucket creation
  - Upload/download
  - Versioning
  - Listing by prefix
- Total data: 100 PB
- Durability: 99.9999% (six nines)
- Availability: 99.99% (four nines)
- Supports both small and large objects

## Capacity Estimation

System design requires concrete numbers.

### Object Size Distribution

Assume:

- 20% small (<1MB), median 0.5MB
- 60% medium (1–64MB), median 32MB
- 20% large (>64MB), median 200MB

### Average Size

(0.2 × 0.5) + (0.6 × 32) + (0.2 × 200)

= 0.1 + 19.2 + 40  
= 59.3 MB per object

---

## Total Objects

Target: 100 PB = 10^11 MB  
Assume 40% utilization

Total objects:
(10^11 × 0.4) / 59.3 ≈ 680 million objects

This immediately implies:

- No single machine can handle this
- Distributed storage is required
- Distributed metadata indexing is required

---

## Metadata Storage

Assume:

- 1KB metadata per object

680 million × 1KB ≈ 680GB metadata

This is manageable but requires sharding due to access patterns.

---

## IOPS Constraints

A typical HDD (~7200 RPM):

- ~100–150 random IOPS

At hundreds of millions of objects:

- Small reads/writes become bottlenecks

Implication:

- Store multiple small objects together in larger files
- Optimize for sequential I/O

---

## Design Philosophy: Separate Metadata from Data

A key design principle:

Metadata and data are stored separately.

This mirrors UNIX file systems.

### UNIX Analogy

- Inode → metadata (name, size, permissions, location)
- Data blocks → actual file content

### Object Storage Mapping

- Metadata store → object info + pointer (UUID)
- Data store → raw bytes

### Why Separate Them?

Because they behave differently:

#### Data

- Immutable
- Large
- Optimized for sequential writes and reads

#### Metadata

- Mutable
- Small
- Requires indexing and fast lookups

They need:

- Different storage engines
- Different scaling strategies
- Different consistency guarantees

Separating them allows independent optimization of each layer.
